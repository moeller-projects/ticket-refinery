User-facing kickstart handoff · As of 2026-07-24 10:40 UTC

Start ticket-refinery safely

A compact handoff for operators and developers: from first configuration to the first validated ADO work item.

Ziel: Eine Queue mit needs-refinement-Items wird verarbeitet. Erfolgreiche Findings werden validiert, kommentiert, angehängt und mit refinement-done markiert. Nicht auflösbare Findings werden mit refinement-blocked markiert.

1. In fünf Minuten loslegen

  1. .env.example nach .env kopieren und ADO-/Pi-Werte setzen.
  2. Mindestens ein repo:<name>-Mapping in src/repos.jsonc eintragen.
  3. Pi-Auth unter ~/.pi/agent/auth.json bereitstellen, falls Subscription-OAuth verwendet wird.
  4. ./check.ps1 ausführen.
  5. ./run.ps1 starten; der Exitcode ist das Betriebssignal.
Copy-Item .env.example .env
# .env bearbeiten: ADO_ORG, ADO_PROJECT, ADO_PAT, Tags, CLONE_DEPTH, PI_MODEL
./check.ps1
./run.ps1
Wichtig: run.ps1 akzeptiert keinen fehlenden .env. Es gibt keinen Fallback auf Platzhalterwerte aus .env.example. Siehe run.ps1:11-23.

2. Was beim Lauf passiert

flowchart TD A[ADO Queue abfragen] --> B{repo:name Tags aufloesbar?} B -- nein --> X[InfraError / Exit 1] B -- ja --> C[Shallow clone in /tmp/refine-id] C --> D[Graphify Index aktualisieren] D --> E[Kommentare + Work Item + Repo-Kontext in Prompt] E --> F[Pi read-only ausfuehren] F --> G{JSON und SourceRefs valide?} G -- nein --> H[Blocked-Kommentar + refinement-blocked] G -- ja --> I{unknowns vorhanden?} I -- ja --> H I -- nein --> J[Attachment + Summary + refinement-done] H --> K[Workspace cleanup] J --> K

Der Graph wird vor Pi erzeugt. Pi soll über den vorbereiteten Kontext argumentieren, nicht das Repository unkontrolliert neu entdecken.

3. Betriebsablauf pro Work Item

8. Übergabe-Checkliste

  • `.env` nicht committen und PAT-Scope minimieren.
  • `src/repos.jsonc` enthält für jedes Work-Item-Repo URL und Branch.
  • ./check.ps1 ist ohne Fehler durchgelaufen.
  • Ein Test-Item wurde entweder refinement-done oder begründet refinement-blocked.
  • Attachment, Kommentar und Tags in ADO manuell geprüft.
  • Bei Codeänderungen exakte SourceRefs/Zeilenbereiche aktualisiert.
Goal: Process a queue of needs-refinement work items. Valid findings are checked, commented, attached, and tagged refinement-done. Unresolvable findings are tagged refinement-blocked.

  2. Add at least one repo:<name> mapping to src/repos.jsonc.
  3. Provide ~/.pi/agent/auth.json when using subscription OAuth.
  4. Run ./check.ps1.
  5. Run ./run.ps1; the exit code is the operational signal.

Important: run.ps1 fails when .env is missing. It does not fall back to placeholder values. See run.ps1:11-23.

2. What happens during a run

flowchart TD A[Query ADO queue] --> B{repo:name tags resolve?} B -- no --> X[InfraError / exit 1] B -- yes --> C[Shallow clone into /tmp/refine-id] C --> D[Refresh Graphify index] D --> E[Comments + work item + repo context into prompt] E --> F[Run Pi read-only] F --> G{JSON and sourceRefs valid?} G -- no --> H[Blocked comment + refinement-blocked] G -- yes --> I{unknowns present?} I -- yes --> H I -- no --> J[Attachment + summary + refinement-done] H --> K[Workspace cleanup] J --> K

The graph is built before Pi runs. Pi should reason over prepared context, not rediscover the repository without bounds.

8. Handoff checklist

  • Do not commit .env; keep PAT scope minimal.
  • src/repos.jsonc has URL and branch for every work-item repository.
  • ./check.ps1 passes.
  • A test item reaches either refinement-done or a justified refinement-blocked.
  • Verify the attachment, comment, and tags in ADO manually.
  • Update exact source references/line ranges when changing code.
