ticket-refinery — onboarding handoff

Audience: new teammate. Goal: run, debug, and change pipeline safely.

What this is

Containerized pipeline that reads ADO work items tagged needs-refinement, clones repos, asks Pi to produce structured findings, validates output, writes back description / acceptance criteria, comments, and moves tags.

Repo map

  • src/refine.py thin orchestrator: env load, DI, queue loop, back-compat process_item
  • src/services/refinement_service.py per-item phase composer (the only seam that touches metrics); splices curated RepositoryContext into the prompt
  • src/services/workspace_service.py clone, cache symlink, Graphify re-index, cleanup
  • src/services/context_service.py existing comments + prompt render (accepts an optional repo_context_section)
  • src/services/publishing_service.py ADO writes (description/AC/title in marker-block, comment, attachment upload, tag transitions)
  • src/ado_client.py ADO REST (WIQL, JSON Patch, comments, attachments)
  • src/git_ops.py parallel shallow clone with per-call PAT credential header
  • src/pi_runner.py Pi CLI subprocess wrapper + JSON parse (raises InfraError on non-JSON)
  • src/validate.py JSONSchema + every sourceRef resolves to a real file (no retry — permanent failure)
  • src/retry.py centralised 3-attempt backoff (1s/2s/4s); opt-in by exception class tuple
  • src/metrics.py counters + timer context manager + immutable MetricsSnapshot
  • src/repository_knowledge.py RepositoryKnowledge facade over GraphifyBackend (default) or FilesystemBackend (fallback)
  • src/repository_context.py RepositoryContext DTO + RepositoryContextBuilder; called by the orchestrator before Pi runs
  • src/prompts/refine.prompt.tmpl.md Pi prompt (curated RepositoryContext spliced in by the orchestrator)
  • src/schema/findings.schema.json output contract (Draft-07)
  • src/repos.jsonc repo registry (repo:<name> → url + defaultBranch)

Dependency direction (no cycles): refine.py → RefinementService → {Workspace, Context, Publishing}Service → {git_ops, pi_runner, ado_client, validate}. Services never import each other. retry, metrics, repository_knowledge, repository_context are independent cross-cutting layers.

Setup

  • Create .env from .env.example
  • Set ADO_ORG, ADO_PROJECT, auth, tags, PI_MODEL
  • Ensure ~/.pi/agent/auth.json exists
  • Use Docker or Podman
  • Graphify ships in the container image (pip install graphifyy); the host doesn't need it

ADO contracts

  • Input tag: needs-refinement
  • Success tag: refinement-done
  • Blocked tag: refinement-blocked
  • Permissions: Work Items Read & Write + Code Read
  • Repo mapping: repo:<name> tags must exist in src/repos.jsonc

Pi integration

Pi runs as a CLI subprocess invoked by src/pi_runner.py:

pi -p <prompt> --model <PI_MODEL>

No --tools flag is set in code — Pi's read-only tool profile is configured by the running image / Pi extension, not by this repo. stdout must be JSON matching src/schema/findings.schema.json; non-JSON raises InfraError (no retry, the parser isn't going to suddenly succeed). Default timeout: PI_TIMEOUT_SECONDS=900.

The application gathers curated repository intelligence (before Pi runs) and splices it into the prompt as a {repo_context} section. The Pi prompt template no longer instructs Pi to invoke any graphify_* tools; it tells Pi to reason over the provided curated content and only use read on the listed files when verifying specific lines. This reduces token spend and prevents Pi from re-running expensive repository-wide searches.

Config reference

All vars are read by refine.Config.from_env(); see .env.example for defaults + inline docs.

  • Azure DevOps: ADO_ORG, ADO_PROJECT, ADO_PAT (or ADO_AUTH_TOKEN / ADO_MCP_AUTH_TOKEN / ADO_API_KEY / SYSTEM_ACCESSTOKEN when PAT is empty — priority order is fixed in code).
  • Tags: TAG_TRIGGER, TAG_DONE, TAG_BLOCKED (must match actual ADO tags).
  • Behaviour: ALLOW_TITLE_EDITS (default false), CLONE_DEPTH, TARGET_LANGUAGE (default English).
  • Pi: PI_MODEL, PI_TIMEOUT_SECONDS (default 900).
  • Container runtime: CONTAINER_ENGINE (docker/podman; auto-detected if blank).

Validation & exit

  • 0 queue drained OR all items reached refinement-done / refinement-blocked
  • 1 infra failure: auth, clone, Pi invocation, unhandled exception
  • 2 launcher failure: missing .env, no container engine

Reaching refinement-blocked (unknowns, validation failure) is a successful run. Only auth / clone / Pi failures fail the container.

Tests

From repo root: python -m pytest -q (use the project's .venv; deps live in requirements.txt, pinned by the uv-built image). Suite covers retry/metrics/repository_knowledge/repository_context units, service-layer orchestration with mocked collaborators, ADO REST with requests mocked, and a back-compat integration of refine.process_item with all externals stubbed. Each leaf module ships a tiny __main__ self-check (run python src/validate.py etc.) so the lazy code is not blind.

Gotchas

  • Inline # comments in .env break parsing (check.ps1 flags them).
  • Podman rootless often fails DNS resolution — run.ps1 forces --dns 8.8.8.8 + 1.1.1.1 on podman.
  • PAT scope: Work Items Read & Write + Code Read. Missing scope → HTTP 401 with HTML body (dumped to /tmp/ado-non-json-response.html).
  • PAT must never end up in a clone URL — git_ops.py uses a short-lived http.extraHeader per clone.
  • Pi subscription OAuth tokens live at ~/.pi/agent/auth.json; run.ps1 mounts it read-write so token refresh works on long runs. Mounting :ro breaks renewal.
  • Missing repo:<name> entry in src/repos.jsonc is a hard fail (raises InfraError), not a guess.
  • Unresolved sourceRef paths never retry — schema/sourceRef failures are permanent; only transient HTTP/subprocess failures are retried.
  • ALLOW_TITLE_EDITS=true lets Pi rewrite the work-item title — off by default, risky.
